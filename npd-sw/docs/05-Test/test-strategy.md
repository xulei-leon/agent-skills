## 1. 测试范围

### 1.1 单元测试 (Unit Test)
- **目标**: 验证每个模块/函数的独立正确性
- **工具**: TBD (Jest / Vitest / Mocha / Pytest)
- **覆盖要求**: 核心逻辑 ≥ 90%，整体 ≥ 80%
- **位置**: `test/ut/`

### 1.2 集成测试 (Integration Test)
- **目标**: 验证模块间协作和数据流转
- **覆盖要求**: 所有接口路径
- **位置**: `test/st/`

### 1.3 系统测试 (System Test)
- **目标**: 验证端到端业务流程
- **覆盖要求**: 所有 FR 需求
- **位置**: `test/st/`

### 1.4 验收测试 (Acceptance Test)
- **目标**: 验证 User Story 验收标准
- **标准**: 每个 US 的 AC 必须有对应的测试用例

---

## 2. 测试流程

```
写 UT → 写代码 → 跑 UT → 通过? ──否──→ 修复代码
   │                                 ↑
   │  (通过)                          │
   └──→ 跑 ST → 通过? ──否──→ 修复代码
            │
         (通过)
            ↓
         Git commit → Review
```

---

## 3. 测试规范

### 3.1 命名规范

| 类型 | 模式 | 示例 |
|------|------|------|
| 单元测试 | `test/ut/m{n}-us{yy}.test.{ext}` | `test/ut/m1-us01.test.ts` |
| 系统测试 | `test/st/st-{id}.test.{ext}` | `test/st/st-001.test.ts` |
| 测试结果 | `test/results/M{n}-test-report.md` | `test/results/M1-test-report.md` |

### 3.2 测试数据

- 测试数据存放: `test/fixtures/`
- 避免硬编码，优先使用工厂模式

### 3.3 测试隔离

- 每个测试独立，不依赖其他测试的执行顺序
- 外部依赖（DB/API）使用 mock 或 test double

---

## 4. 质量门禁

- UT 通过率 100% 方可提交
- ST 通过率 100% 方可进入评审
- 测试覆盖率低于阈值需补充用例
